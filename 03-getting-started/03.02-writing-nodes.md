# Writing Nodes

A good node is useful to both humans and agents. A bad node is noise.

## NODE.md: two jobs

Every NODE.md does exactly two things:

```
┌─────────────────────────────────┐
│           NODE.md               │
│                                 │
│  1. Describe the domain         │
│     What is this folder about?  │
│     One paragraph. Be clear.    │
│                                 │
│  2. Declare the owner           │
│     Who controls this folder?   │
│     In the frontmatter.         │
└─────────────────────────────────┘
```

Good NODE.md:

```yaml
---
title: "Backend"
owners: [alice]
---

# Backend

Backend architecture, API design, and data access patterns.
Alice owns all decisions in this subtree.
```

Bad NODE.md:

```yaml
---
title: "Backend"
owners: [alice]
---

# Backend

This is the backend folder.
```

The bad one tells you nothing an agent can't already see from the folder name.

## Leaf nodes: record decisions, not just facts

A leaf node should capture _why_, not just _what_.

Good leaf node:

```markdown
# Why PostgreSQL

We chose PostgreSQL over MongoDB because:
- Our data is relational. Joins matter.
- We need ACID transactions for billing.
- The team has 3 years of Postgres experience.

We considered MongoDB for its flexible schema
but decided the tradeoff was not worth it for our use case.
```

Bad leaf node:

```markdown
# Database

We use PostgreSQL.
```

The bad one records a fact. The good one records the decision, the reasoning, and the alternatives considered. An agent reading the good one can make informed decisions about the database. An agent reading the bad one knows nothing useful.

## Frontmatter: keep it minimal

Only include fields that are needed.

```yaml
# Always include:
title: "Why PostgreSQL"
owners: [alice]

# Include when relevant:
status: approved              # for decisions
soft_links: [/product/data]   # for cross-references

# Include on member nodes:
role: "Engineer"
domains: ["backend"]
```

Do not add fields for the sake of completeness. If a field does not help a human or agent understand the node, leave it out.

## One decision per node

Do not put three decisions in one file. Split them.

```
Bad:
  engineering/decisions.md     ← 12 decisions in one file

Good:
  engineering/why-monolith.md
  engineering/why-postgres.md
  engineering/why-rest-over-grpc.md
```

Each node is a single, findable, ownable unit. An agent looking for the database decision should find one file, not scroll through twelve.

## Write for the agent that arrives tomorrow

Your nodes will be read by agents that do not exist yet. They will arrive at your tree with zero context and try to understand your organization by walking the tree.

Ask yourself: if a new agent starts at this node, can it understand what this domain is about and why things are the way they are?

If yes, the node is good. If no, add context.
