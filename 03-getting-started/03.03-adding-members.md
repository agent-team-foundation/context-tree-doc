# Adding Members

Every participant on the team gets a node under `members/`. Humans and agents.

## Three types of members

```
/members/
├── NODE.md
├── alice.md           ← human
├── bob.md             ← human
├── agent-alice.md     ← personal agent (alice's assistant)
├── agent-bob.md       ← personal agent (bob's assistant)
└── agent-kael.md      ← autonomous agent (independent)
```

**Humans.** Set direction, make judgment calls, grant permissions.

**Personal agents.** Work on behalf of a specific human. They hold delegated permissions. Agent-alice can approve things alice has authorized it to approve.

**Autonomous agents.** Independent agents that own domains and run continuously. They are accountable to the team, not to any individual human.

## Creating a human member

```bash
cat > members/alice.md << 'EOF'
---
title: "alice"
owners: [alice]
role: "Engineer"
domains:
  - "backend"
  - "infrastructure"
---

## About

Backend engineer. Focused on API design
and service reliability.

## Current Focus

- Auth system v2 redesign
- Migrating storage to PostgreSQL
EOF
```

## Creating an agent member

```bash
cat > members/agent-kael.md << 'EOF'
---
title: "agent-kael"
owners: [agent-kael]
role: "Agent"
domains:
  - "documentation"
  - "code review"
---

## About

Autonomous agent that maintains the documentation
subtree and reviews backend PRs.

## Current Focus

- Keeping docs in sync with code changes
- Reviewing PRs that touch /engineering/
EOF
```

Same format. Same rules. An agent's node looks like a human's node.

## Creating a personal agent

```bash
cat > members/agent-alice.md << 'EOF'
---
title: "agent-alice"
owners: [alice]
role: "Personal Agent"
domains:
  - "alice's delegated tasks"
---

## About

Alice's personal agent assistant.
Holds delegated permissions from alice.

## Permissions

- Can approve PRs on /engineering/backend/
- Can draft but not merge on /product/
EOF
```

Note: the owner is `alice`, not `agent-alice`. Alice controls her own agent's node.

## After creating a member

Assign them as owner of relevant nodes:

```yaml
# /engineering/backend/NODE.md
---
title: "Backend"
owners: [alice, agent-kael]
---
```

Now alice and agent-kael both own the backend subtree. Either one can approve changes (OR logic).

## Keep nodes fresh

Your personal node is how other members understand you. When your focus changes, update your node. Stale information leads to bad decisions.

An agent checking your node before sending you a request will read your current focus. If it is outdated, the agent makes wrong assumptions about what you are working on.
